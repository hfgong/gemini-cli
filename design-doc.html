1.5 Codebase Search & Discovery

Syntactic Search (Current)

ToolImplementation
grep_searchripgrep binary (auto-downloaded to ~/.gemini/bin/)
globnpm glob package, recency-sorted results
read_fileDirect fs read with line range support
lsBFS directory traversal, 200-item limit

No embeddings, no AST parsing, no vector search. All search is exact-match regex or glob pattern.

Codebase Investigator Sub-agent

agents/codebase-investigator.ts

An autonomous LLM sub-agent (not an index) that reasons through the codebase using read-only tools. Returns structured JSON:

{
  "SummaryOfFindings": "...",
  "ExplorationTrace": ["step1", "step2", ...],  // BLOATS main context
  "RelevantLocations": [
    { "FilePath": "...", "KeySymbols": [...] }
  ]
}

Max 3 min, 10 turns. Uses Flash model with extended thinking.

Key finding: Claude Code also uses agentic grep+glob search (no vector DB). A 2026 Amazon Science paper confirmed keyword search achieves >90% of RAG-level performance for code. The retrieval strategy itself is not a competitive gap — the gap is in how efficiently context is utilized.

Part 2: Gap Analysis & Proposed Changes

2.1 System Prompt Changes (Autonomy & Instruction Following)

All changes below are text modifications in packages/core/src/prompts/snippets.ts. No code logic changes required.

Figure 5 — Prompt Change Map (red = change targets)

Change A: Flip Inquiry/Directive Default P0 — Highest Impact

snippets.ts:222 renderCoreMandates()

Problem: The prompt says "Assume all requests are Inquiries unless they contain an explicit instruction." This causes the model to default to passive analysis, explaining instead of acting. When a user says "fix the login bug", the model may just analyze it rather than fix it.

BEFORE
Assume all requests are Inquiries unless
they contain an explicit instruction to
perform a task.
AFTER
Assume requests are Directives when they
describe a desired outcome, a problem to
fix, or a change to make. Treat requests
as Inquiries only when they explicitly ask
for explanation, analysis, or advice (e.g.,
"how does X work?", "explain Y", "what do
you think about Z?").

Change B: Reduce Over-Cautious Confirmation P1

snippets.ts:556-559 mandateConfirm()

Problem: When a user reports a bug, the agent asks "Should I fix this?" instead of fixing it. Claude Code would just fix it.

BEFORE (interactive)
Do not take significant actions beyond
the clear scope of the request without
confirming with the user. If the user
implies a change (e.g., reports a bug)
without explicitly asking for a fix,
**ask for confirmation first**.
AFTER (interactive)
Stay within the scope of the request but
interpret user intent generously. If the
user reports a bug or describes a problem,
proceed to fix it unless they explicitly
say they only want analysis. Ask for
confirmation only when the intended action
is genuinely unclear or when multiple
significantly different approaches exist.

Change C: Soften "Explain Before Acting" P2

snippets.ts:229 renderCoreMandates()

Problem: Forces a text response before every tool call, adding latency and risking the model stopping after explaining rather than acting.

BEFORE
Never call tools in silence. You MUST
provide a concise, one-sentence explanation
of your intent or strategy immediately
before executing tool calls.
AFTER
When beginning a new phase of work or
changing strategy, briefly state your
intent. For sequential tool operations
within an established plan, tool calls
may proceed without narration.

Change D: Strengthen Persistence Through Errors P1

snippets.ts:223 renderCoreMandates() — Proactiveness

Problem: Current instruction is buried and weak. The model gives up after 1-2 failed approaches.

BEFORE
When executing a Directive, persist
through errors and obstacles by
diagnosing failures in the execution
phase and, if necessary, backtracking...
AFTER
When executing a Directive, you MUST
persist through errors and obstacles.
Do not give up or ask the user for help
unless you have tried at least 3 different
approaches. Diagnose failures, backtrack
to research if needed, and try alternative
solutions. Never tell the user "I was
unable to complete this" without having
exhausted viable alternatives. If a tool
fails, analyze why, fix the issue, and
retry.

Change E: Add Task Completeness Mandate P2

snippets.ts:324 renderPrimaryWorkflows() — after Validation paragraph

Problem: Model finishes one subtask and stops, waiting for user acknowledgment before continuing.

BEFORE
(no instruction about completing all subtasks)
AFTER (add after line 324)
**Task Completeness:** Execute ALL
identified subtasks before presenting
results to the user. Do not stop after
completing individual subtasks to ask
for feedback unless you are genuinely
blocked. A task is only finished when
all subtasks are complete and validated.

Change F: Add Self-Sufficiency Mandate NEW P0

snippets.ts:228 renderCoreMandates() — new bullet

Problem: Users report Gemini says "you can do X by running Y" instead of just doing it. There's no instruction against this.

BEFORE
(no such instruction exists)
AFTER (add as new bullet)
- **Self-Sufficiency:** When given a
Directive, execute the work yourself using
available tools. Never instruct the user to
perform steps manually that you can
accomplish with your tools (e.g., don't say
"you can run npm install" — just run it).
The user chose an agent to do the work,
not to receive instructions.

2.2 Context Management Changes

Figure 6 — Context Pipeline: Current vs Proposed

Change G: Raise Compression Threshold P0 — 1 line change

chatCompressionService.ts:40

Problem: Compression triggers at 50% of token limit, discarding useful context when only half the window is used. Claude Code compacts at 95%.

- const DEFAULT_COMPRESSION_TOKEN_THRESHOLD = 0.5;
+ const DEFAULT_COMPRESSION_TOKEN_THRESHOLD = 0.75;

Impact: +262k tokens of usable context before compression kicks in. Gemini models handle 1M tokens natively, so this is safe.

Change H: Trim Codebase Investigator Output P1

codebase-investigator.ts:25-45, 92

Problem: The ExplorationTrace array (every tool call logged) flows into the main context, bloating it. The main agent only needs findings and locations.

  // Current: processOutput sends everything including ExplorationTrace
  processOutput: (output) => JSON.stringify(output, null, 2),

  // Proposed: Strip ExplorationTrace, keep only actionable data
  processOutput: (output) => {
    const { ExplorationTrace, ...actionableData } = output;
    return JSON.stringify(actionableData, null, 2);
  },

Impact: Reduces sub-agent context footprint by ~40-60% while keeping all actionable information.

Change I: Enhance Compression Prompt Fidelity P2

snippets.ts:737-815 getCompressionPrompt()

Problem: The XML state_snapshot format loses specific file paths, error messages, and tool results. After compression, the agent "forgets" what it discovered.

Proposed enhancements to the compression prompt:

  <!-- Add to key_knowledge instructions -->
  <!-- You MUST preserve ALL specific file paths that were read,
       modified, or found to be relevant. Include line numbers
       where specific issues were identified. -->

  <!-- Add to artifact_trail instructions -->
  <!-- Preserve exact error messages and stack traces encountered
       during tool execution. Include the specific tool call that
       produced each error and the resolution applied. -->

  <!-- Add new section -->
  <failed_approaches>
      <!-- Approaches that were tried and failed, with specific
           reasons for failure. This prevents the agent from
           repeating failed strategies after compression. -->
  </failed_approaches>

4.2 Testing Strategy

Automated Tests

# Unit tests (must pass)
nvm use 20 && npm run test

# Full preflight (lint + format + tests)
npm run preflight

# Integration tests
npm run test:e2e

Existing snapshot tests for prompt composition will need updating for changed text.

Manual Validation Scenarios

  1. Autonomy: "Fix the TypeScript errors in src/" — should act without asking
  2. Instruction following: "The button is misaligned" — should fix, not explain
  3. Self-sufficiency: "Add a test for the login function" — should write test, not instruct user
  4. Persistence: Give a task with a failing build — should retry with fixes
  5. Context: Long session with 20+ tool calls — verify compression happens later
  6. Inquiry guard: "How does the auth module work?" — should explain, NOT modify code
Important: The CONTRIBUTING.md notes that changes to Context Efficiency prompts (snippets.ts:176-215) require running SWE-bench benchmarks before committing. Our changes to other sections (Mandates, Workflows) don't fall under this specific warning, but running SWE-bench on the compression threshold change (G) would be ideal if the benchmarking infrastructure is available.

4.3 Suggested PR Strategy

PR 1 (Prompt — Autonomy): Changes A, F, B, D, E, C — all in snippets.ts

PR 2 (Context — Compression): Change G — single constant in chatCompressionService.ts

PR 3 (Context — Subagent): Change H — investigator output trimming

PR 4 (Context — Compression Prompt): Change I — enhanced compression prompt

Ship PRs 1 & 2 first for maximum impact. PRs 3 & 4 can follow independently.
