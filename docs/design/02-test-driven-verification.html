Design Doc 02: Test-Driven Verification

Force the agent to verify its work by running tests before declaring completion.
Priority: Highest ROI Risk: Low Effort: Medium

1. Problem Statement

Observation from FeatureBench A/B Testing

In our FeatureBench lite evaluation (28 tasks), neither the baseline nor the changed agent ever ran pytest to verify its work. Both versions wrote code and immediately called complete_task without any validation.

The one task the baseline resolved (pandas/iceberg) was correct by luck — the changed version produced nearly identical code but got one parameter wrong (scan.limit() vs table.scan(limit=)), which a single test run would have caught.

Gemini's "laziness problem" manifests most critically at the end of the task: the agent writes code and declares victory without checking if the code actually works. This is the single highest-leverage intervention point because:

  • It doesn't increase context during the creative/coding phase (no reasoning quality degradation)
  • It catches both trivial errors (syntax, imports) and semantic errors (wrong API usage)
  • It gives the agent a chance to self-correct before the final patch is captured
  • The test infrastructure already exists in every SWE-bench/FeatureBench task

2. Proposed Design

2.1 Approach: Completion Gate

Intercept the complete_task tool call and inject a verification step. The agent must demonstrate that its changes pass relevant tests before the completion is accepted.

2.2 Implementation Options

Option A: Prompt-Only (Lowest effort)

Add explicit instructions to the system prompt requiring the agent to run tests before calling complete_task.

// Add to snippets.ts TASK_COMPLETENESS section:

Before calling complete_task, you MUST:
1. Identify test files related to your changes (look for test_*.py, *_test.py, or tests/ directory)
2. Run the relevant tests: e.g., `python -m pytest path/to/test_file.py -x -q`
3. If tests fail, fix your code and re-run until they pass
4. Only call complete_task after tests pass

If you cannot find relevant tests, state this explicitly and explain
how you verified correctness manually.

Pros: Zero code changes, easy to A/B test
Cons: Gemini may ignore the instruction (it already ignores "be thorough"); increases system prompt size

Option B: Completion Gate Hook (Medium effort, recommended)

Modify processFunctionCalls() in local-executor.ts to intercept complete_task and inject a verification turn.

// In local-executor.ts::processFunctionCalls()
// When complete_task is detected:

if (toolName === TASK_COMPLETE_TOOL_NAME) {
  // Before accepting completion, inject verification
  const verificationResult = await this.runVerificationStep(chat, signal);

  if (verificationResult.testsFound && !verificationResult.allPassed) {
    // Reject completion, tell the agent to fix failures
    taskCompleted = false;
    // Feed test failure output back to the model
    const feedback = `Your changes have test failures:\n${verificationResult.output}\n` +
      `Please fix the failing tests and try again.`;
    // Continue the loop with this feedback
  }
}

Pros: Enforced — agent can't skip verification; structured feedback loop
Cons: Requires code changes; may increase task duration; needs test discovery heuristic

Option C: Pre-completion Tool (Higher effort)

Add a new verify_changes tool that the agent can (and should) call before complete_task.

// New tool: verify_changes
{
  name: "verify_changes",
  description: "Run tests related to your changes and report results. " +
    "Call this BEFORE complete_task to validate your work.",
  parameters: {
    test_command: "Command to run tests (e.g., pytest path/to/test.py)",
    working_directory: "Directory to run from"
  },
  execute: async (params) => {
    const result = await shell.execute(params.test_command, {
      cwd: params.working_directory,
      timeout: 120_000
    });
    return {
      passed: result.exitCode === 0,
      output: result.stdout + result.stderr,
      exitCode: result.exitCode
    };
  }
}

Pros: Clean separation of concerns; agent retains agency over what to test
Cons: Agent may still skip calling it; needs prompt changes to encourage use

2.3 Recommended Approach: A + B Combined

Use Option A (prompt changes) to encourage the agent to test proactively, and Option B (completion gate) as a safety net to catch cases where the agent skips testing.

3. Test Discovery Heuristic

The completion gate needs to find relevant test files. Heuristic:

async function discoverRelevantTests(changedFiles: string[]): Promise<string[]> {
  const testFiles: string[] = [];

  for (const file of changedFiles) {
    const dir = path.dirname(file);
    const base = path.basename(file, path.extname(file));

    // Pattern 1: test_foo.py for foo.py
    const candidates = [
      path.join(dir, `test_${base}.py`),
      path.join(dir, `${base}_test.py`),
      path.join(dir, 'tests', `test_${base}.py`),
      path.join(dir, '..', 'tests', `test_${base}.py`),
    ];

    for (const candidate of candidates) {
      if (await fileExists(candidate)) {
        testFiles.push(candidate);
      }
    }
  }

  // Fallback: grep for imports of changed modules in test files
  if (testFiles.length === 0) {
    // Use glob to find test files, grep for module references
  }

  return [...new Set(testFiles)];
}

5. Risk Assessment

RiskSeverityMitigation
Test runs consume time, reducing turns available for coding Medium Set test timeout (120s). Only run on completion, not every turn. Count verification turns separately from coding turns.
Test output floods context, degrading Gemini's reasoning Medium Truncate test output to first 50 lines of failures. Summarize: "3/10 tests failed: test_foo, test_bar, test_baz"
Wrong test files discovered, causing false negatives Low Use heuristic discovery + let agent override with explicit test command. Don't hard-block on test failures — warn and allow override after 3 attempts.
Infinite fix-test loop if tests are flaky or unrelated Low Cap verification retries at 3. After 3 failures, accept completion with a warning.

6. Expected Impact

Based on FeatureBench lite analysis:

  • pandas/iceberg — Would have caught the scan.limit() vs table.scan(limit=) error. One test run = task resolved. (+1 resolved)
  • seaborn/algorithms — Changed version scored 8/13 instead of 12/13. Test feedback could guide fixes for the remaining 5 tests.
  • xarray/treenode — Changed version scored 11/34 instead of 17/34. Test output would help identify regressions.
  • 8 newly-attempted tasks — All produced patches but 0 resolved. Test verification would either fix them or prevent bad patches.

Conservative estimate: +5-15% absolute improvement in resolve rate, primarily from catching near-miss solutions.

The RALPH Loop pattern (proven in production)

An autonomous testing framework where the agent iteratively executes tests, verifies results, and fixes failures:

  1. Run test suite
  2. If test fails and fix_attempts < 3: analyze root cause, implement fix, re-test
  3. After 3 failures: mark as "known_issue" and move on

Real-world result: 38 test cases completed in 3.5 hours with 100% pass rate, catching bugs that code review alone missed.

Reference: RALPH Loop for Claude Code

Test output handling: keep it simple

JetBrains NeurIPS 2025 research compared observation masking (simple truncation) vs LLM summarization of test output. Finding: simple truncation matches or beats summarization while being cheaper. LLM summarization actually caused agents to run 13-15% longer by obscuring stop signals.

Practical recommendation: truncate test output to first 50 lines of failures. Write full output to a file if the agent needs more detail.

Reference: JetBrains: "The Complexity Trap"

mini-swe-agent: proof that minimal scaffolding works

The most striking finding: mini-swe-agent achieves >74% on SWE-bench Verified with ~100 lines of Python. No tool-calling interface, no plugins, no retrievers. Just bash commands via subprocess.run and a linear message history. The simplicity is the feature.

This validates Option A (prompt-only) as a viable approach — with a strong enough model, you just need to tell it to run tests. The question is whether Gemini is "strong enough" to follow the instruction reliably.

Reference: mini-swe-agent GitHub

8. Competitor Analysis: How Other CLIs Handle Verification

Trae Agent (ByteDance) — Mandatory 7-Step Workflow

Trae's system prompt (agent_prompt.py) enforces a rigid 7-step methodology where steps 3 and 6 are explicitly marked as "crucial":

  1. Understand the Problem
  2. Explore and Locate
  3. Reproduce the Bug (Crucial Step) — "CREATE A SCRIPT that reliably reproduces the bug"
  4. Debug and Diagnose
  5. Develop and Implement
  6. Verify and Test Rigorously — run reproduction script, run test suite, write new pytest cases
  7. Summarize Work

This is essentially our Option A (prompt-only) with much more specificity. The prompt doesn't just say "run tests" — it prescribes the exact sequence: reproduce first, then fix, then verify with multiple methods.

Trae Agent — Completion Gate with Patch Validation

Trae implements exactly our Option B. In trae_agent.py (lines 236-244), _is_task_completed() checks if the patch is empty and returns the agent to work if so:

if self.llm_indicates_task_completed(llm_response):
    if self._is_task_completed(llm_response):  # Validates patch exists
        execution.agent_state = AgentState.COMPLETED
    else:
        # Send error back as user message — forces agent to continue
        return [LLMMessage(role="user", content=self.task_incomplete_message())]

Additionally, Trae strips test file modifications from patches (lines 205-226), filtering paths matching /test/, /tests/, test_, tox.ini. This prevents the agent from "solving" tasks by modifying test expectations rather than fixing source code.

Grok CLI — No Explicit Test Verification

Grok CLI has no test verification gate or test-before-completion prompt. It relies entirely on the model's judgment about when to call task complete. This makes Grok a useful baseline — it represents what happens without verification infrastructure.

However, Grok does have a useful pattern: headless mode (--prompt flag) with JSON output that auto-approves all confirmations. This is useful for benchmark automation and could be adapted for running verification steps non-interactively.

9. Deeper Analysis: Why Verification Works Disproportionately Well

The Asymmetry of Generation vs. Verification

There's a fundamental asymmetry in LLM capabilities: verification is dramatically easier than generation. An LLM that can only solve 30% of coding tasks from scratch can often judge the correctness of a solution with 80%+ accuracy when given test results. This is why test-driven verification has such outsized impact:

  • Generation requires world knowledge — the model must know APIs, conventions, edge cases
  • Verification only requires reading — the test output tells the model exactly what went wrong
  • Iterative fixing narrows the search space — each test failure eliminates a class of wrong solutions

The implication: even if Gemini is a weaker generator than Claude, a test feedback loop can close a significant portion of the quality gap. The model doesn't need to get it right the first time — it needs to recognize and fix its own errors.

The Reproduce-First Pattern

Trae's emphasis on "reproduce the bug BEFORE fixing it" is more than good engineering practice. It has a specific cognitive benefit for LLMs:

  • Writing a reproduction script forces the model to understand the bug — it can't reproduce what it doesn't understand
  • The reproduction script becomes the verification oracle — the model now has a concrete yes/no signal, not just "does the code look right?"
  • It's a cheaper test than running the full suite — a single-file script runs in seconds, while a full pytest suite might take minutes

For Gemini specifically, the reproduce-first pattern addresses laziness from a different angle: instead of telling the model "be thorough," it gives it a concrete, small, achievable task (write a 10-line reproduction script) that naturally leads to deeper understanding.

Test Output as Context — A Double-Edged Sword

Feeding test output back to the model creates a tension specific to Gemini:

  • Benefit: Test output is the highest-information-density context possible — it tells the model exactly what's wrong
  • Risk: A full pytest traceback with 500 lines of stack traces degrades Gemini's reasoning (context fragility)
  • Solution: Truncate aggressively. JetBrains showed that simple truncation beats LLM summarization. But we can be smarter than blind truncation:
// Smart test output formatting:
// 1. Summary line: "3 failed, 7 passed, 2 errors in 4.2s"
// 2. For each failure: test name + assertion error (not full traceback)
// 3. Only show full traceback for the FIRST failure
// Total: ~20-50 lines regardless of how many tests fail

This is a micro-optimization that compounds: each verification cycle stays lightweight, enabling more fix-test iterations within the same context budget.

Why Patch Filtering Matters More Than It Seems

Trae's test-file exclusion filter seems like a benchmark anti-gaming measure, but it encodes a deeper principle: the agent should only modify source code, never test expectations. Without this:

  • A lazy model will "fix" a failing test by changing the expected value rather than fixing the code
  • In FeatureBench/SWE-bench, this produces patches that pass the FAIL_TO_PASS tests trivially but break semantic correctness
  • This is Gemini's most likely failure mode given its laziness tendency

We should implement this filter in Gemini CLI's complete_task handler, independent of the completion gate.
