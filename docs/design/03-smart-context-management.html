Design Doc 03: Smart Context Management

Address Gemini's context fragility through surgical context loading, observation masking, and repository-aware indexing.
Priority: Critical Risk: Medium Effort: High

1. The Core Problem: Context Fragility

"Context is the bottleneck, not intelligence." — Anthropic, Spotify, and JetBrains research teams independently

Gemini's reasoning quality degrades predictably as context grows, even within its stated 1M-token window. This is not unique to Gemini — it's a well-documented phenomenon called "context rot" (Chroma Research) and the "Lost in the Middle" effect (Stanford/Berkeley):

  • Performance is highest when relevant info is at the beginning or end of context
  • Information in the middle of long contexts is effectively invisible
  • Multi-hop reasoning (connecting facts across multiple locations) degrades much faster than simple retrieval
  • Counterintuitively, shuffled text performs better than coherent text because coherent text creates stronger positional bias

This explains Gemini's "laziness": the model has learned that loading more context makes it dumber, so it avoids exploration as a self-preservation strategy. Our Phase 1 prompt changes fight this instinct, causing the agent to load more context but reason less accurately.

References: Chroma: Context Rot | Lost in the Middle (arXiv) | Context Length Alone Hurts LLM Performance

3. Proposed Changes for Gemini CLI

3.1 Observation Masking (Low effort, high impact)

What

After each turn, truncate tool output observations older than N turns while preserving the full action/reasoning history. JetBrains research (NeurIPS 2025) showed this matches or beats LLM summarization while being simpler and cheaper.

Why it works for Gemini specifically

A typical SWE agent's context is ~80% tool output (file contents, command results) and ~20% reasoning. Masking old outputs cuts context by ~50% without removing any of the agent's reasoning chain. Since Gemini's reasoning degrades with context size, this directly addresses the root cause.

Implementation

// In chatCompressionService.ts or a new observationMasker.ts

interface ObservationMaskingConfig {
  // Keep full output for the last N turns
  recentWindowSize: number;  // default: 10
  // For older turns, replace output with a summary placeholder
  summaryTemplate: string;   // e.g., "[Read {filename}: {lineCount} lines]"
  // Never mask outputs smaller than this
  minSizeToMask: number;     // default: 500 chars
}

function maskOldObservations(
  history: ChatMessage[],
  config: ObservationMaskingConfig
): ChatMessage[] {
  const recentCutoff = history.length - config.recentWindowSize;

  return history.map((msg, i) => {
    if (i >= recentCutoff) return msg;  // Keep recent turns intact
    if (msg.role !== 'tool') return msg; // Keep user/assistant messages

    const output = msg.content;
    if (output.length < config.minSizeToMask) return msg;

    // Replace with compact placeholder
    return {
      ...msg,
      content: generatePlaceholder(msg),
      _originalLength: output.length,
    };
  });
}

Affected files

  • packages/core/src/services/chatCompressionService.ts — Add masking before compression
  • packages/core/src/agents/local-executor.ts — Apply masking in turn loop

3.2 Repository Map via Tree-Sitter (Medium effort, high impact)

What

Aider's most innovative feature: parse the entire codebase with tree-sitter, extract all function/class/method signatures, build a dependency graph, run PageRank to identify the most-referenced symbols, and include only the top-ranked signatures in context. The LLM gets a "table of contents" of the entire repo without loading full file contents.

Why it works for Gemini specifically

Instead of Gemini blindly reading whole files (which bloats context), the repo map gives it a navigational overview. The agent can then surgically request only the specific functions it needs. This turns Gemini's large context window into an advantage — it can hold a broad repo map plus a few deep-dived functions, rather than a few random full files.

Design

Implementation approach

Gemini CLI already has a codebase investigator sub-agent. We can enhance it with tree-sitter:

  1. Add web-tree-sitter as a dependency (works in Node.js)
  2. On first tool call, parse all source files in the project root
  3. Build a signature index: Map<filePath, Definition[]>
  4. Inject the top-ranked signatures into the system prompt (refreshed on file changes)
  5. Cache the index; re-parse only changed files (use file mtime)

Affected files

  • packages/core/src/agents/codebase-investigator.ts — Enhance with tree-sitter
  • packages/core/src/services/repoMapService.ts — New: build and maintain repo map
  • packages/core/src/prompts/snippets.ts — Inject repo map into system prompt

Reference: Aider: Repository Map | Building a better repository map with tree-sitter

3.3 Hierarchical File Localization (Medium effort, high impact)

What

Inspired by Agentless (41.2% SWE-bench Verified at $0.70/issue) and LocAgent (92.7% file accuracy): before the agent starts coding, run a lightweight localization phase that narrows from "entire repo" to "specific functions to edit."

Three-phase funnel

  1. File-level: Show repo structure + repo map to LLM, ask "which 5 files are most relevant?"
  2. Symbol-level: For each candidate file, show only the skeleton (class/function declarations). Ask "which functions need modification?"
  3. Context loading: Load only the identified functions (plus immediate dependencies) into context

This is what the existing codebase-investigator sub-agent partially does, but it loads full files and returns verbose traces. The improvement is to make it return only function-level pointers, and load context surgically.

Affected files

  • packages/core/src/agents/codebase-investigator.ts — Add skeleton-based narrowing
  • packages/core/src/tools/read-file.ts — Add function-extraction mode (read only a specific function by name)

Reference: Agentless Paper | LocAgent: Graph-Guided Localization

3.4 Enhanced Compression with Observation Awareness (Low effort)

What

Our Phase 1 Change I added <failed_approaches> tracking to the compression prompt. Extend this with observation-aware compression:

  • When compressing, always preserve: tool call names + arguments, error messages, file paths mentioned
  • Always discard: full file contents (replace with "[Read src/foo.py: 200 lines, contains class Foo, Bar]"), verbose command output, duplicate information
  • Preserve the 5 most recently accessed files in full (Claude Code's approach)

Affected files

  • packages/core/src/services/chatCompressionService.ts — Structured compression rules
  • packages/core/src/prompts/snippets.ts — Enhanced compression prompt

6. Gemini-Specific Considerations

Gemini has a genuine 1M-token context window, which is larger than Claude's 200K. This creates a unique opportunity:

  • Repo map can be larger — Include more signatures than Aider/Claude can fit
  • But don't fill it — The "Lost in the Middle" effect means quality peaks around 50-100K tokens of relevant content, not 1M of everything
  • Use the space for structure, not content — A comprehensive repo map + targeted function reads is better than dumping whole files
  • Observation masking is extra important — Without it, Gemini's context fills with stale tool output, leaving less room for the repo map and active work

7. Competitor Analysis: How Other CLIs Handle Context

Trae Agent — Codebase Knowledge Graph (CKG)

Trae has a working implementation of what we're proposing in 3.2 (repo map). Their ckg_tool.py uses tree-sitter + SQLite:

  • Parses Python, Java, C/C++, TypeScript, JavaScript using tree-sitter
  • Indexes into SQLite: functions, classes, methods with line numbers and parent relationships
  • Exposes three search commands: search_function, search_class, search_class_method
  • Returns actual function bodies, not just signatures
  • Persistent across calls — state maintained in memory, no re-parsing

Key difference from Aider's repo map: Trae's CKG is a tool the agent queries (pull model), while Aider's repo map is injected into the system prompt (push model). The pull model is more context-efficient — the agent only loads what it needs — but requires the model to know when and how to search.

Trae Agent — Lakeview Trajectory Summarization

Trae's lake_view.py provides trajectory-aware context management. Each agent step is tagged semantically (WRITE_TEST, VERIFY_TEST, EXAMINE_CODE, WRITE_FIX, etc.) and summarized by an LLM. This serves two purposes:

  • Monitoring: Human observers can see what the agent is doing at a glance
  • Context compression: The tagged summaries could replace raw history in the context window (though Trae doesn't currently use them this way)

The semantic tagging is the novel part — rather than generic "summarize the last N turns," the tags create a structured activity log that's both human-readable and machine-parseable. This could improve our compression prompt (3.4) by making compression tag-aware.

Grok CLI — Fuzzy File Search + Smart Truncation

Grok's search tool (search.ts) has two ideas worth borrowing:

  • Fuzzy file matching: Scores results by exact match (100), substring (80), path match (60), fuzzy (10-40). This helps the model find files even when it doesn't know the exact name — a common problem for Gemini which often guesses wrong file paths.
  • Smart truncation in view_file: Files >N lines show only the first 10 lines + "... +N more lines" indicator. The agent must explicitly request ranges. This is a lightweight version of SWE-agent's ACI window concept (Design Doc 04, section 4).

Grok CLI — Morph Fast Apply (Abbreviated Edits)

Grok optionally integrates with Morph API for code editing at 4,500+ tokens/sec. The interesting idea is the abbreviated edit format:

// Instead of sending full file content for replacement:
function processUser(user) {
  // ... existing code ...
  user.name = sanitize(user.name);  // NEW: sanitize input
  // ... existing code ...
}

The // ... existing code ... comments let the model express edits without reproducing the entire file. A separate, faster model expands these into full edits. This is context-efficient — the LLM only generates the changed lines. For Gemini, where output token cost is high and context is precious, this pattern could significantly reduce both cost and context usage.

8. Deeper Analysis: The Context Management Hierarchy

Why "More Context" Doesn't Mean "Better Context"

Our Phase 1 results proved this empirically: pushing Gemini to explore more (+25% patch rate) degraded accuracy (-3.2% F2P). The fundamental insight is that context management operates on three orthogonal axes:

  1. Volume — How many tokens are in context (reduce this)
  2. Relevance — What fraction of context tokens are useful for the current decision (increase this)
  3. Placement — Where in the context window are the important tokens (beginning and end, not middle)

Most solutions focus on axis 1 (reduce volume). The breakthrough solutions — repo maps, CKG, hierarchical localization — focus on axis 2 (increase relevance). Almost nobody addresses axis 3, but it matters enormously for Gemini due to "Lost in the Middle."

Push vs Pull: Two Philosophies of Context Loading

The competitor analysis reveals two fundamentally different approaches:

Push ModelPull Model
HowPre-load info into system prompt / contextAgent queries tools to fetch what it needs
ExamplesAider repo map, Cursor embeddings, our compression summariesTrae CKG, Gemini codebase-investigator, grep/glob tools
ProsAgent always has the info; no risk of forgetting to searchOnly loads what's needed; context-efficient
ConsWastes context on info that may not be neededAgent must know to search; Gemini is lazy about searching

The optimal solution for Gemini is a hybrid: push a lightweight repo map (structure/navigation) and pull detailed content (function bodies) on demand. The pushed map is small enough to not degrade reasoning, but provides enough scaffolding that the agent knows what to pull.

The Real Bottleneck: Not Context Size, but Information Density

Consider two scenarios with the same token count (50K):

  • Scenario A: 50K tokens = 3 full files (2000 lines each, lots of boilerplate, imports, docstrings)
  • Scenario B: 50K tokens = repo map (2K) + 10 targeted function extracts (3K each) + test output (5K) + reasoning history (13K)

Scenario B has 10x higher information density — every token serves a purpose. This is why Agentless achieves 41% on SWE-bench at $0.70/issue: it doesn't load more context, it loads better context through hierarchical localization.

For Gemini specifically, high-density context also mitigates the "Lost in the Middle" effect. When every token matters, there's less filler for important information to get lost in.

Observation Masking: A Counter-Intuitive Win

The JetBrains finding that simple truncation beats LLM summarization seems counter-intuitive — shouldn't a summary preserve more information? The explanation lies in how LLMs use context:

  • LLMs don't need to "remember" old tool outputs — they've already acted on them
  • The actions themselves (what tool was called, what file was read, what command was run) encode more useful information than the outputs
  • LLM-generated summaries are lossy in unpredictable ways — they may drop the one detail that matters later
  • Simple truncation with a placeholder ("[Read src/foo.py: 200 lines]") preserves the fact that the file was read without the content, which is usually enough

The deeper principle: preserve the narrative arc (what was tried and why) but discard the raw data (tool outputs). This is essentially what experienced developers do when explaining a debugging session — they describe the steps, not the log contents.
