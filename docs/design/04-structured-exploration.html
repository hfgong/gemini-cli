Design Doc 04: Structured Exploration & Backtracking

Multi-hypothesis exploration with explicit state management and recovery from wrong directions.
Priority: High Risk: High Effort: High

1. The Problem: Tunnel Vision

When Gemini encounters a complex task, it tends to commit to its first hypothesis and tunnel deeper — even when that hypothesis is wrong. It lacks the ability to:

  • Recognize when it's going in the wrong direction
  • Backtrack cleanly to a known-good state
  • Try alternative approaches without accumulated context pollution from failed attempts

This is especially damaging in SWE-bench/FeatureBench tasks where the first approach is often wrong — the agent needs to explore the codebase, form hypotheses, test them, and pivot.

2. Industry Approaches

Linear Agent (Current)

Single trajectory. If the approach fails, the agent tries to fix it in-place, accumulating error context.

Used by: Current Gemini CLI, basic ReAct agents

Problem: Context pollution from failed attempts degrades subsequent reasoning

SWE-agent ACI

Constrains agent to a 100-line viewing window with scroll/search navigation. Linter gates syntactically incorrect edits. undo_edit enables reversal.

Used by: SWE-agent (~37% SWE-bench)

Insight: Constraints help — preventing the agent from seeing too much at once improves focus

Tree Search (CodeTree / SWE-Search)

MCTS over solution paths. Git commits as state snapshots. A critic/value agent scores each branch. Can explore multiple approaches in parallel.

Used by: CodeTree (Salesforce), SWE-Search

Result: +23% relative improvement over linear agents. Weaker models with tree search nearly match stronger models without it.

EnCompass (MIT)

Clones program runtime at annotated "branchpoints." Explores paths in parallel via MCTS or beam search. Automatic backtracking on failure.

Used by: Research (NeurIPS 2025)

Result: +15-40% accuracy improvement at 16x LLM call budget. 82% less code than hand-implemented search.

Agentless Multi-Patch

Generate N candidate patches independently (no agent loop). Use test execution to select the best one. No backtracking needed because each attempt starts fresh.

Used by: Agentless (41% SWE-bench at $0.70)

Insight: Sometimes the cheapest backtracking is "don't track — just restart"

Context-Window Reset

When a strategy fails, start a completely fresh context window with zero traces of the failed attempt. The next strategy starts clean.

Used by: Various agents

Insight: For Gemini's context fragility, a fresh start may outperform carrying forward error context

References: CodeTree | SWE-Search | EnCompass (MIT) | SWE-agent ACI

3. Proposed Design: Git-Based Checkpointing

The most practical approach for Gemini CLI, balancing impact with implementation complexity:

3.1 Core Mechanism: Checkpoint & Rollback

Use git as the state management layer. Before the agent starts a new approach, create a checkpoint (commit). If the approach fails (tests don't pass after N attempts), roll back to the checkpoint and try a different approach with a fresh context summary.

// New: packages/core/src/agents/checkpoint-manager.ts

class CheckpointManager {
  private checkpoints: Map<string, string> = new Map(); // name -> commitSha

  async createCheckpoint(name: string, workdir: string): Promise<string> {
    // Stage all changes and create a checkpoint commit
    await exec('git add -A', { cwd: workdir });
    await exec(`git commit -m "checkpoint: ${name}" --allow-empty`, { cwd: workdir });
    const sha = await exec('git rev-parse HEAD', { cwd: workdir });
    this.checkpoints.set(name, sha.trim());
    return sha.trim();
  }

  async rollback(name: string, workdir: string): Promise<void> {
    const sha = this.checkpoints.get(name);
    if (!sha) throw new Error(`No checkpoint: ${name}`);
    await exec(`git reset --hard ${sha}`, { cwd: workdir });
  }

  async listCheckpoints(): Promise<string[]> {
    return [...this.checkpoints.keys()];
  }
}

3.2 Integration with Agent Loop

3.3 Fresh Context on Backtrack

The key insight for Gemini

When rolling back, don't just undo the code changes — also reset the conversation context. Start a new context window with:

  1. The original task description
  2. The repo map (from Design Doc 03)
  3. A summary of what was tried and why it failed
  4. Explicit instruction to try a different approach

This prevents the "context pollution" problem where traces of a failed approach bias the model toward making the same mistake again.

// Context for attempt #2 after rollback:

const freshContext = `
## Task
${originalTaskDescription}

## Repository Map
${repoMap}

## Previous Attempt (FAILED — do NOT repeat this approach)
Approach: ${attempt1.summary}
Failure reason: ${attempt1.failureReason}
Key test errors: ${attempt1.testErrors.slice(0, 5).join('\n')}

## Instructions
Try a DIFFERENT approach. Do not repeat the failed strategy.
Consider: ${suggestedAlternatives}
`;

3.4 Lightweight Alternative: Best-of-N Sampling

The Agentless approach — simpler, effective

Instead of complex tree search, run the entire agent N times independently (N=3-5) with different random seeds or slight prompt variations. Use test results to select the best patch. This is what Agentless does, and it's surprisingly competitive.

Advantages for Gemini:

  • Each attempt gets a completely fresh context — no context fragility
  • No backtracking logic needed — just run N independent processes
  • Can run in parallel if API rate limits allow
  • Natural diversity in approaches from LLM sampling

Disadvantages:

  • N times the API cost
  • Requires test execution to select the best patch (see Design Doc 02)
  • Doesn't help with tasks where the fundamental approach is always wrong

4. The SWE-agent Insight: Constraints Help

One of the most counterintuitive findings from SWE-agent research: constraining what the agent can see improves performance. Their Agent-Computer Interface (ACI) limits the agent to a 100-line window at a time, with explicit scroll/search commands.

This is directly relevant to Gemini's context fragility:

  • Current read_file: Returns the entire file. A 2000-line file dumps 2000 lines into context.
  • ACI-style read_file: Returns a 100-line window. Agent must explicitly navigate to find what it needs.

This trades off convenience for focus. The agent loads less irrelevant context, so what it does load gets more attention from the model.

Practical adaptation for Gemini CLI

Rather than fully adopting SWE-agent's ACI (which would be a major UX change), add a smart truncation mode for read_file:

  • For files < 200 lines: return full content (current behavior)
  • For files > 200 lines: return the function/class containing the target line, plus 50 lines of surrounding context
  • Always show the file's skeleton (all function/class definitions with line numbers) so the agent knows what's in the file

6. Risk Assessment

RiskSeverityMitigation
Git operations interfere with the user's repo state High Only use checkpointing in benchmark/headless mode, or work on a detached branch. Never modify the user's working branch without consent.
Multiple attempts multiply API costs Medium Cap at 3 attempts per task. Use lighter model for initial exploration if available.

7. Recommendation

Start with the prompt-level approach:

  1. Add instructions telling the agent to checkpoint its approach mentally and consider alternatives if stuck
  2. Implement Best-of-N sampling as a command-line option (--attempts 3) — simple, effective, no code complexity
  3. Add smart file truncation to read_file — low risk, immediate context savings

Only pursue full git-based checkpointing if the simpler approaches don't show enough improvement. The research suggests Best-of-N is surprisingly competitive with tree search for most tasks.

8. Competitor Analysis: How Other CLIs Handle Exploration

Trae Agent — Sequential Thinking Tool

Trae provides a sequential_thinking_tool.py that gives the agent a structured interface for multi-step reasoning:

  • Linear progression: Numbered thoughts (1, 2, 3...) with a declared total
  • Revision: Can go back and revise thought #2 after writing thought #5
  • Branching: Can fork from any previous thought to explore an alternative
  • Dynamic expansion: Can add more thoughts even after reaching the declared total
  • Hypothesis tracking: Thoughts can be tagged as hypotheses for later verification

This is essentially a structured scratchpad that makes the agent's reasoning explicit and backtrackable. The branching mechanism directly addresses the "tunnel vision" problem: instead of committing to hypothesis #1, the agent can fork at a decision point and explore both branches.

Key insight: this doesn't require any code changes to the agent loop. It's just a tool with memory (the thought history) that gives the model a framework for organizing its own reasoning.

Trae Agent — Reflection on Failed Tools

In base_agent.py (lines 246-257), Trae explicitly reflects failed tool results back as assistant-role messages:

"The tool execution failed with error: {error}.
Consider trying a different approach or fixing the parameters."

This is subtly different from just returning the error as a tool result. By framing it as guidance ("consider a different approach"), it nudges the model toward strategic pivoting rather than blind retry. The message comes from the assistant role, which models tend to treat as more authoritative than tool output.

Grok CLI — Fuzzy Edit Matching

Grok's text editor (text-editor.ts, lines 411-454) solves a practical exploration problem: when the model's mental model of the file diverges from reality.

  • When exact string match fails for multi-line edits, Grok falls back to structural analysis
  • It tokenizes by control flow keywords and compares structure, not text
  • This allows edits to succeed even when whitespace, comments, or formatting differ from what the model predicted

This addresses a specific Gemini failure mode: the model reads a file, does some work, then tries to edit the file but its recalled version doesn't exactly match (especially after context compression). Fuzzy matching turns a hard failure into a successful edit.

Grok CLI — Session-Based Approval Acceleration

Grok's confirmation service lets users approve "all operations of this type" once. This is relevant for automated exploration: when running benchmarks or autonomous tasks, per-operation approval creates friction that slows the agent and fills context with approval dialogues. Session-based approval eliminates this overhead.

Gemini CLI already has -y for auto-approve, but the per-type granularity (approve file ops but require confirmation for bash) is a useful UX pattern for production use.

9. Deeper Analysis: The Nature of Agent Exploration

The Exploration-Exploitation Tradeoff in Coding Agents

Coding agents face a classic explore-exploit dilemma, but with a twist specific to LLMs:

  • Exploration (reading files, searching code, testing hypotheses) — adds context, which degrades reasoning for context-fragile models
  • Exploitation (writing code based on current understanding) — risks being wrong if exploration was insufficient

Claude (lower context fragility) can afford deep exploration. Gemini cannot. This explains Gemini's laziness as a rational strategy: given its context fragility, premature exploitation is often better than thorough exploration that pollutes context.

The solution isn't to force Gemini to explore more — it's to make exploration cheaper in context terms. Every technique in this design doc (checkpointing, fresh context, smart truncation) serves this goal: enabling exploration without the context tax.

Why Tree Search Is Probably Overkill (For Now)

CodeTree and EnCompass show impressive gains (+23%, +15-40%) from MCTS-style tree search. But the implementation complexity is enormous, and the gains are relative to a weak baseline. Consider:

  • Best-of-N with N=3 captures most of the benefit of tree search (diverse attempts, best patch selection) with zero architectural complexity
  • Linear agent + test verification (Design Doc 02) already provides one "branch" per test failure cycle
  • Trae's sequential thinking tool gives lightweight branching without any agent loop changes

The marginal value of tree search over these simpler approaches is small for Gemini, because Gemini's bottleneck is context quality, not search strategy. A tree search that explores 5 branches in the same context window will likely produce 5 mediocre solutions rather than 1 good one. Best-of-N with fresh context for each attempt is strictly better for a context-fragile model.

Structured Thinking: Tool vs Prompt

Trae implements structured thinking as a tool. Claude uses extended thinking as a model capability. The tool-based approach has a unique advantage for weaker models: it forces the model to externalize reasoning into a structured format rather than doing it implicitly.

For Gemini, a structured thinking tool could work as follows:

// Before implementing a fix, the agent would call:
sequential_thinking({
  thought: "Hypothesis: The bug is in the data serialization layer because...",
  thought_number: 1,
  total_thoughts: 3,
  next_action: "Read src/serialization.py to verify"
})

// After reading the file:
sequential_thinking({
  thought: "Hypothesis confirmed. The issue is in serialize_date() at line 45...",
  thought_number: 2,
  total_thoughts: 3,
  next_action: "Write fix and reproduction test"
})

// This creates an auditable reasoning chain that:
// 1. Forces the model to articulate its hypothesis before acting
// 2. Creates natural checkpoints for backtracking
// 3. Prevents "stream of consciousness" exploration that wastes context

The key insight: structured thinking is cheap in context (50-100 tokens per thought) but forces expensive reasoning to happen before expensive actions. For a model prone to laziness, this front-loads the thinking that would otherwise be skipped.

The "Fresh Context" Advantage for Weak Models

A pattern emerging across all competitor analysis: weaker models benefit disproportionately from fresh context. Consider the evidence:

  • Agentless: No agent loop at all — each LLM call gets perfectly curated context. 41% SWE-bench.
  • Best-of-N: Each attempt starts fresh. Often matches tree search with complex state management.
  • mini-swe-agent: Linear history only (no compression). 74% SWE-bench.

The pattern: the simpler the context management, the better weaker models perform. Complex context (compressed summaries, long tool histories, accumulated state) creates noise that degrades reasoning. Fresh, clean context with only the essential information produces better results.

This suggests our most impactful exploration strategy for Gemini is not sophisticated tree search, but rather: try, test, and if stuck, restart with a clean slate + failure summary. Simple, radical, effective.

Practical Priority: What to Implement First

Based on the competitor analysis and deeper thinking, the revised priority order is:

  1. Sequential thinking tool (from Trae) — Minimal code, forces better reasoning, no context overhead. Can be a prompt-only change that tells Gemini to think step-by-step in a structured format.
  2. Best-of-N with fresh context — Simple --attempts N flag. Each attempt starts clean. Use test results to select best patch. No backtracking logic needed.
  3. Fuzzy edit matching (from Grok) — Prevents hard failures when Gemini's memory of file contents drifts. Low effort, high resilience improvement.
  4. Git checkpointing — Only if Best-of-N proves insufficient. The complexity is warranted only if we need mid-attempt rollback (which Best-of-N avoids entirely).
